# Workflow Fragment Production-Readiness Testing Plan

## Goal

Make the workflow engine credible for production use by testing it as a durable, database-backed,
crash-recoverable state machine under realistic and hostile conditions.

The current test suite and refreshed smoke scripts are a good start, but they are **not yet enough**
to call the engine production-ready. Production readiness requires a repeatable matrix that runs the
same scenarios against **Postgres, MySQL, and SQLite**, uses deterministic custom workflows,
exercises mocked external services, and intentionally kills/restarts processes and databases at
unsafe times.

## Production-readiness bar

A release is production-ready only when all of these are true:

- The full smoke suite runs against fresh Postgres, MySQL, and SQLite databases.
- Every run creates an isolated database/schema/file and tears it down or archives it with logs.
- Multiple durable-hook dispatchers can run concurrently without duplicate effects.
- App processes, dispatchers, mock service processes, and databases can be killed/restarted during
  active workflow steps without corrupting state.
- External-service calls are mocked, deterministic, and fault-injected: latency, 500s, timeouts,
  malformed JSON, partial responses, and idempotent retries.
- The suite includes custom workflows intentionally designed to stress the runner, not just the
  example app's demo workflows.
- The suite records evidence: commands, env, database backend, seeds, instance IDs, terminal counts,
  attempt counts, consumed events, and logs.
- Known unsupported behavior is explicitly documented and guarded by tests.

## Database matrix

Every baseline, stress, and fault scenario should run against this matrix:

| Backend  | Fresh target per run                                          | Must cover                                                                                              |
| -------- | ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| Postgres | new database or schema, e.g. `fragno_wf_smoke_<runId>`        | migrations, concurrent dispatchers, DB restart, row locks/OCC, JSON payloads, timestamp/cursor behavior |
| MySQL    | new database, e.g. `fragno_wf_smoke_<runId>`                  | migrations, JSON handling, timestamp precision/time zones, deadlock/retry behavior, connection loss     |

## Custom smoke workflows to build

Add a dedicated smoke app/harness with workflows that are deterministic and intentionally hostile:

- **Approval workflow:** sequential waits, duplicate/out-of-order events, pause/resume/terminate.
- **Large payload workflow:** large params, event payloads, step results, and final output.
- **Deterministic retry workflow:** fails exactly N times, then succeeds; separate variant exhausts
  retries.
- **Backoff workflow:** constant, linear, and exponential backoff with controlled clock assertions.
- **Sleep/wake workflow:** sleep, sleepUntil, early wake hooks, late wake hooks.
- **Wait-timeout workflow:** event before timeout, after timeout, at boundary, and duplicate events.
- **Long-running workflow:** blocks inside a step long enough to kill app/dispatcher/DB mid-step.
- **Parallel workflow:** many `Promise.all` branches, mixed success/retry/failure, stable step keys.
- **Race workflow:** `Promise.race`/`Promise.any` with losers that finish late and must not
  overwrite.
- **Nested workflow:** nested steps with repeated names at different depths to verify identity.
- **External-service workflow:** calls a local mock HTTP service from inside `step.do`.
- **Step transaction workflow:** uses `tx.mutate`, `tx.serviceCalls`, `tx.onTerminalError`,
  `tx.emit`, `tx.previousEmissions`, and `tx.onEvent`.
- **Emission workflow:** emits ordered stream payloads while running, fails/retries, then cleans up.
- **Schema-validation workflow:** invalid params/output must fail predictably.
- **High-cardinality workflow:** creates many instances/events to validate pagination/list filters.

## Mock external service requirements

Build a local mock service process used by smoke workflows. It must support deterministic faults via
run ID and request key:

- success with stable JSON;
- fixed latency and jitter;
- HTTP 500/429/408;
- connection reset / socket close;
- timeout / never responds;
- malformed JSON;
- large response body;
- idempotency key recording so retries can be checked;
- endpoint to dump all calls for invariant checks.

The smoke suite should kill and restart this mock service while workflows are waiting on it.

## 3. Correctness matrix

| Area               | Required invariants                                                                                                                            |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| Instance lifecycle | status transitions are monotonic toward terminal except pause/resume; terminal states are never overwritten                                    |
| Events             | user events persist in order; only one event is consumed per `waitForEvent`; late events after timeout do not wake timed-out waits             |
| Retry              | attempts never exceed `maxAttempts`; `NonRetryableError` skips retry; retryable failures schedule `nextRetryAt`                                |
| Time               | sleep/wait hooks do not complete early; clock skew does not permanently stall work                                                             |
| Parallel steps     | sibling/nested step keys are stable; completed siblings are replay-skipped; race losers cannot overwrite winners                               |
| Step tx            | mutations/service calls/emissions commit exactly once and only on successful step boundary unless `onTerminalError` applies                    |
| Pause/terminate    | paused instances do not consume user events; termination wins against in-flight ticks and queued hooks                                         |
| Emissions          | current-step emissions are visible during active steps, ordered per epoch/sequence, and cleaned up after completion/error                      |
| API                | route validation produces declared errors; list pagination has no duplicates/gaps; history is a complete ordered snapshot                      |
| Durability         | process crashes, dispatcher crashes, app restarts, mock-service restarts, and DB restarts recover without partial commits or duplicate effects |
| Migrations         | repeated/concurrent migrations converge on one schema version without duplicate indexes on Postgres, MySQL, and SQLite                         |
| Cross-database     | same logical scenario produces equivalent workflow state on Postgres, MySQL, and SQLite                                                        |
| Clients            | mutators invalidate affected list/detail/history hooks and never import server-only modules into browser bundles                               |

## 6. Obsolete old-smoke assumptions

The old scripts referenced a public restart route, `runNumber`, workflow task rows, history cursors,
and `_runner/tick`. The current source does not expose those contracts. The updated smoke suite now
uses durable hook dispatchers, public instance-list cursors, full history snapshots, and terminal
cleanup/integrity checks instead.
