Pi workflows are durable TypeScript workflows built on the Workflows fragment. Use them when a Pi
session needs more than the interactive chat loop: multiple agent steps, tool-result branching,
parallel reviewers, races against user input, or explicit approvals.

Register custom workflows with the same `createPi()` builder that registers agents and tools.

```ts
import { createPi, definePiWorkflow } from "@fragno-dev/pi-fragment";

const pi = createPi()
  .withAgent("researcher", researcher)
  .withAgent("writer", writer)
  .withWorkflow(researchWriteApprove)
  .build();
```

Create sessions by selecting the workflow and passing schema-validated input.

```json
{
  "workflow": "research-write-approve",
  "input": { "topic": "durable LLM workflows" }
}
```

## Sequential research → write → approve

Use explicit message passing between agent steps. Each agent call returns a new result; later steps
choose which previous messages to continue from.

```ts
import { z } from "zod";
import { definePiWorkflow } from "@fragno-dev/pi-fragment";

export const researchWriteApprove = definePiWorkflow(
  {
    name: "research-write-approve",
    schema: z.object({ topic: z.string() }),
  },
  async (ctx) => {
    const research = await ctx.agent("researcher").prompt("research", {
      input: { text: `Research ${ctx.params.topic}.` },
    });

    const draft = await ctx.agent("writer").prompt("draft", {
      input: { text: "Write a concise answer using the research." },
      messages: research.messages,
    });

    const approval = await ctx.waitForEvent("approval", {
      allowed: ["prompt", "complete"],
    });

    if (approval.kind === "prompt") {
      const revision = await ctx.agent("writer").prompt("revise", {
        input: approval.input,
        messages: draft.messages,
      });
      return { messages: revision.messages };
    }

    return { messages: draft.messages };
  },
);
```

## Decision tool → branch to an agent

Use `definePiTool(...)` when workflow code needs typed structured tool details. Passing a handoff
tool to `stopOnTools` stops the agent run after that tool result, so the workflow can branch.

```ts
import { Type } from "typebox";
import { z } from "zod";
import { definePiTool, definePiWorkflow } from "@fragno-dev/pi-fragment";

const classifyRequest = definePiTool({
  name: "classify_request",
  description: "Classify an incoming support request",
  parameters: Type.Object({ request: Type.String() }),
  handoff: true,
  resultSchema: z.object({
    kind: z.enum(["bug", "feature", "question"]),
    confidence: z.number(),
  }),
  async execute(_toolCallId, params) {
    return {
      content: [{ type: "text", text: "Request classified." }],
      details: { kind: "question", confidence: 0.9 },
      terminate: true,
    };
  },
});

export const triageWorkflow = definePiWorkflow(
  { name: "triage-request", schema: z.object({ request: z.string() }) },
  async (ctx) => {
    const triage = await ctx.agent("triage").prompt("triage", {
      input: { text: ctx.params.request },
      stopOnTools: [classifyRequest],
    });

    const classification = triage.toolCalls(classifyRequest).latest();

    switch (classification.details.kind) {
      case "bug":
        return ctx.agent("debugger").prompt("debug", {
          input: { text: ctx.params.request },
          messages: triage.messages,
        });
      case "feature":
        return ctx.agent("planner").prompt("plan", {
          input: { text: ctx.params.request },
          messages: triage.messages,
        });
      case "question":
        return ctx.agent("support").prompt("answer", {
          input: { text: ctx.params.request },
          messages: triage.messages,
        });
    }
  },
);
```

## Parallel reviewers → merge

Wrap concurrent work in a named parent `ctx.step.do(...)` step. The parent gives replay a stable
outer boundary, and each branch still has a stable child step name.

```ts
import { z } from "zod";
import { definePiWorkflow } from "@fragno-dev/pi-fragment";

export const parallelReviewWorkflow = definePiWorkflow(
  { name: "parallel-review", schema: z.object({ draft: z.string() }) },
  async (ctx) => {
    const [security, clarity, correctness] = await ctx.step.do("parallel-reviews", async () =>
      Promise.all([
        ctx.agent("security-reviewer").prompt("security-review", {
          input: { text: ctx.params.draft },
        }),
        ctx.agent("clarity-reviewer").prompt("clarity-review", {
          input: { text: ctx.params.draft },
        }),
        ctx.agent("correctness-reviewer").prompt("correctness-review", {
          input: { text: ctx.params.draft },
        }),
      ]),
    );

    return ctx.agent("editor").prompt("merge-reviews", {
      input: { text: "Merge reviewer feedback into a final answer." },
      messages: [...security.messages, ...clarity.messages, ...correctness.messages],
    });
  },
);
```

`Promise.race(...)` and `Promise.any(...)` follow the same rule: put the combinator inside a named
parent step and put all durable branch work inside that callback.

## Replay rules

<Callout type="warn" title="Keep workflow structure deterministic">
  Workflow replay depends on stable step structure. Do not build step names from random IDs, current
  time, or partial streamed LLM output.
</Callout>

- Step names should be string literals or derived from already completed durable data.
- Agent steps do not mutate shared workflow-level message state. Pass `messages` explicitly and use
  returned `messages` when composing later steps.
- Concurrent `Promise.all`, `Promise.race`, and `Promise.any` calls must be nested under a named
  parent `ctx.step.do("stable-name", ...)` step.
- Handoff tools should set `handoff: true` and usually return `terminate: true`; pass them through
  `stopOnTools` when workflow code needs the first structured decision result.
